# CodexPill Seal Adapter Classification

This note classifies the reusable-looking CodexPill Seal adapter plumbing after the feature-owned recorder split. It is intentionally CodexPill-side only; moving code into Seal is out of scope for RGR-197.

## Decisions

| Surface                                    | Decision                                                       | Reason                                                                                                                                                                                                                                                                                                    |
| ------------------------------------------ | -------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CodexPillSealProofSession`                | Propose to Seal, keep local until then                         | This is the only generic runtime adapter shape: register one feature, start a live run, record events and snapshots, finish, and cancel. CodexPill should not expand it with product behavior. If another Apple client needs the same lifecycle wrapper, move it into Seal or a Seal-owned Apple adapter. |
| `CodexPillSealProofRecorderFactory`        | Keep in CodexPill for now                                      | It reads CodexPill validation environment keys and maps legacy CodexPill scenario names into feature-owned recorders. That compatibility and app bootstrap policy are CodexPill-owned.                                                                                                                    |
| `CodexPillSealProofRecorderRegistry`       | Keep in CodexPill for now                                      | The registry is a local composition seam between the factory and feature-owned scenario catalogs. It should stay small and only answer whether a feature owns a legacy CodexPill scenario.                                                                                                                |
| `CodexPillSealProofRecorder`               | Simplify only when more feature recorders appear; keep for now | The facade is local dependency-injection plumbing for menubar instrumentation. It should not grow scenario declarations, event APIs, or generic recorder behavior.                                                                                                                                        |
| Feature recorder registry shape            | Keep in CodexPill for now                                      | `AccountSealProofRecorderRegistry` and `HostSealProofRecorderRegistry` translate legacy scenario names into feature scenario catalogs, then return feature-facing recorders. Scenario meaning, invariant IDs, and evidence timing remain feature-owned.                                                   |
| Environment/run activation parsing         | Keep in CodexPill for now                                      | `CODEXPILL_VALIDATION_SCENARIO` and `CODEXPILL_SEAL_PROOF_OUTPUT` are CodexPill validation launcher contracts. Seal should own generic runner flags only after a Seal runner exists.                                                                                                                      |
| Artifact/report/result consumption helpers | No CodexPill adapter helper to promote in this layer           | Current shared validation code produces Seal proof artifacts but does not consume Seal result or report artifacts. Report/result consumption belongs to validation scripts or future Seal-owned result APIs, not this adapter layer.                                                                      |

## Local Guardrails

- `Features/Validation` stays UI-surface-neutral and must not import AppKit, SwiftUI, or MenuBar snapshot types.
- CodexPill tests should cover the app obligations: environment activation, legacy scenario routing, feature selection, emitted event names, sanitized payloads, and proof artifact paths.
- CodexPill tests should not assert Seal verifier internals such as expectation rule encoding, rule composition, or manifest schema details beyond the minimal feature/scenario and evidence references CodexPill relies on for integration.
- Feature recorders keep CodexPill semantics local: account and host scenario IDs, invariant IDs, evidence DTOs, event timing, and fixture-facing compatibility names.
